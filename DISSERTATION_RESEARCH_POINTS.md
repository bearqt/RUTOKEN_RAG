# Исследовательские Направления Для Магистерской Диссертации

Тема диссертации: **«Проектирование корпоративного ИИ-ассистента»**.

Ниже перечислены направления научно-исследовательской работы, которые можно обосновать на материале реализованного в проекте RAG для технической базы знаний Рутокен.

## 1. Структурный chunking технической документации

Суть направления:
исследование влияния способа разбиения документации на чанки на качество retrieval и generation.

Что можно сравнивать:
- фиксированные чанки по символам;
- chunking по заголовкам;
- chunking по таблицам;
- function-level chunking для reference-страниц PKCS#11;
- смешанный структурный chunking для prose, tables, code и API reference.

Исследовательский вопрос:
как структура документа влияет на качество поиска и итогового ответа корпоративного ИИ-ассистента.

Ожидаемая ценность:
доказательство того, что для корпоративных баз знаний структурный chunking превосходит наивное разбиение по длине.

## 2. Metadata enrichment как способ повышения качества retrieval

Суть направления:
исследование влияния дополнительных метаданных на релевантность поиска.

Какие метаданные уже используются в проекте:
- `products`;
- `interfaces`;
- `os_tags`;
- `language_tags`;
- `api_symbols`;
- `components`;
- `doc_family`.

Что можно сравнивать:
- retrieval без enrichment;
- retrieval с базовыми metadata;
- retrieval с domain-specific metadata;
- retrieval с entity-aware boosting.

Исследовательский вопрос:
в какой степени структурные и доменные признаки улучшают качество поиска в корпоративном RAG.

## 3. Domain-specific NER и entity normalization

Суть направления:
исследование подходов к извлечению и нормализации сущностей в технической документации.

В проекте особенно важны сущности:
- модели устройств Рутокен;
- интерфейсы `PKCS#11`, `PC/SC`, `CryptoAPI`, `CNG`;
- компоненты `KeyBox`, `ЦУР`, `rtEngine`, `OpenSC`;
- API-символы `C_*`, `CKR_*`, `CKA_*`;
- платформы и языки программирования.

Что можно сравнивать:
- отсутствие NER;
- alias-based NER;
- regex-based extraction;
- hybrid alias + regex;
- LLM-based entity extraction;
- комбинированный подход.

Исследовательский вопрос:
какой метод entity extraction наиболее эффективен для технического enterprise-корпуса.

## 4. Hybrid retrieval для корпоративного ИИ-ассистента

Суть направления:
исследование комбинированного поиска по смыслу и по точным совпадениям.

Что можно сравнивать:
- dense-only retrieval;
- sparse-only retrieval;
- dense + sparse fusion;
- hybrid retrieval + reranking;
- adaptive retrieval в зависимости от типа запроса.

Исследовательский вопрос:
какой тип retrieval лучше подходит для смешанных enterprise-запросов, где одновременно важны и семантика, и exact match.

## 5. Стратегии fusion в retrieval pipeline

Суть направления:
исследование методов объединения dense и sparse выдачи.

Что можно сравнивать:
- линейное объединение score;
- Reciprocal Rank Fusion;
- rank-based fusion;
- query-dependent fusion;
- adaptive weighted fusion.

Исследовательский вопрос:
как стратегия fusion влияет на качество поиска по технической документации.

## 6. Роль cross-encoder reranking в enterprise RAG

Суть направления:
оценка вклада этапа reranking в итоговое качество поиска.

Что можно сравнивать:
- retrieval без reranker;
- rerank top-10;
- rerank top-20;
- rerank top-50;
- влияние reranker на overview-вопросы;
- влияние reranker на function-level API вопросы.

Исследовательский вопрос:
насколько reranker улучшает итоговую релевантность retrieval в корпоративном ИИ-ассистенте.

## 7. Query analysis и query rewriting

Суть направления:
исследование предобработки пользовательского запроса перед retrieval.

Что можно сравнивать:
- отсутствие query rewriting;
- heuristic rewrite;
- LLM-based rewrite;
- rule-based intent-to-symbol mapping;
- извлечение фильтров из запроса;
- комбинированный query analysis.

Исследовательский вопрос:
как переписывание запроса и извлечение фильтров влияют на retrieval quality.

## 8. Специализированный retrieval для API/reference-документации

Суть направления:
разделение стратегий поиска для различных типов документации.

В корпусе явно различаются:
- обзорные статьи;
- how-to инструкции;
- таблицы совместимости;
- reference-страницы функций и API.

Что можно исследовать:
- единый retrieval mode;
- routing по типу запроса;
- отдельный reference-mode для API-функций;
- эвристическое и model-based определение типа запроса.

Исследовательский вопрос:
нужно ли адаптировать retrieval strategy под жанр документа и тип пользовательского вопроса.

## 9. Детерминированная оценка качества RAG

Суть направления:
разработка и анализ benchmark-методики без использования `LLM-as-judge`.

В проекте уже реализованы проверки:
- `required_terms`;
- `required_any`;
- `forbidden_terms`;
- `citations`;
- `exact_match`;
- `refusal`.

Что можно исследовать:
- устойчивость deterministic evaluation;
- корреляцию deterministic metrics с экспертной оценкой;
- пригодность rule-based benchmark для enterprise RAG.

Исследовательский вопрос:
можно ли надежно оценивать корпоративный RAG без judge-модели.

## 10. Faithfulness и controllability корпоративного ИИ-ассистента

Суть направления:
исследование механизмов, которые уменьшают галлюцинации и делают ответы управляемыми.

Что можно исследовать:
- влияние строгого system prompt;
- влияние citations;
- влияние ограничения ответа только retrieved context;
- влияние отказа от ответа при отсутствии релевантного контекста.

Исследовательский вопрос:
какие механизмы наиболее эффективны для повышения достоверности ответов корпоративного ИИ-ассистента.

## 11. Компромисс между качеством, latency и стоимостью

Суть направления:
исследование trade-off между качеством ответа и вычислительными затратами.

Что можно сравнивать:
- с rewrite и без rewrite;
- с reranker и без reranker;
- разные значения `top-k`;
- разные candidate count;
- разные генеративные модели;
- кэширование query analysis и embeddings.

Исследовательский вопрос:
какая конфигурация pipeline обеспечивает наилучший баланс между качеством, скоростью и стоимостью.

## 12. Актуальность знаний и переиндексация

Суть направления:
исследование обновления индекса и поддержания актуальности корпоративной базы знаний.

Что можно исследовать:
- полную переиндексацию;
- инкрементальную переиндексацию;
- детекцию изменений в документах;
- влияние устаревших данных на качество ответа;
- обработку конфликтующих версий знаний.

Исследовательский вопрос:
как организовать поддержку актуальности корпоративного ИИ-ассистента при изменении базы знаний.

## 13. Объяснимость ответов и доверие пользователя

Суть направления:
исследование способов объяснения ответа корпоративного ИИ-ассистента.

Что можно исследовать:
- влияние URL-цитирования на доверие;
- section-level citations;
- показ retrieved chunks пользователю;
- краткое объяснение, почему выбран именно этот источник.

Исследовательский вопрос:
какая форма объяснимости наиболее полезна для пользователей корпоративной системы.

Суть направления:
